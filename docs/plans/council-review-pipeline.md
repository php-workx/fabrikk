# Council Review Pipeline

**Status:** Draft — awaiting annotation review
**Date:** 2026-03-14
**Branch:** `feature/council-review-pipeline`

---

## 1. Problem

The current `attest tech-spec review` performs only structural checks (required section headings present). It does not evaluate content quality, completeness, security implications, testability, or domain correctness. The technical spec is the single source of truth for everything downstream — execution plans, task compilation, worker instructions — so its quality is the ceiling for the entire pipeline.

## 2. Goal

Replace the structural-only review with a council-based review pipeline that produces a well-rounded, multi-perspective technical specification through:

1. Automated generation of review personas tailored to the spec's content and technology stack.
2. Parallel execution of persona-driven reviews using existing CLI tools.
3. A judge/editor consolidation pass that cross-validates findings and updates the spec.
4. Two fixed review rounds with full traceability of accepted and rejected findings.
5. Human final review with `@@` annotation support for remaining issues.

## 3. Flow

```
ROUND 1:
  spec v0 (human draft)
  → judge analyzes spec, generates review personas
  → parallel persona reviews (with nudge pass per reviewer)
  → judge/editor consolidates:
      - cross-validates findings for accuracy
      - applies valid findings to produce spec v1
      - records rejected findings with explanations
  → outputs: spec v1, rejection-log-r1.json, individual review files

ROUND 2:
  spec v1
  → judge re-analyzes spec, regenerates domain personas (fixed personas stay)
  → parallel persona reviews (see spec v1 + round 1 findings for context)
  → judge/editor consolidates again
  → outputs: spec v2, rejection-log-r2.json, individual review files

HUMAN REVIEW:
  human receives: spec v2, rejection logs r1+r2, all individual review outputs
  → human uses @@ annotations for remaining issues
  → unresolved @@ annotations block tech-spec approval
  → human approves → spec enters existing execution-plan pipeline
```

## 4. Review Personas

### 4.1 Fixed Personas

These run in every review regardless of the spec's content:

| Persona | Focus | Perspective |
|---------|-------|-------------|
| **Security & Performance Engineer** | Threat surface, data flows, resource bounds, scaling bottlenecks, permission models | "What can go wrong under adversarial or high-load conditions?" |
| **Testability & Test Design Reviewer** | Acceptance criteria clarity, test boundaries, coverage gaps, deterministic verification, edge cases | "Can every requirement be verified? Are the acceptance scenarios sufficient?" |

### 4.2 Dynamic Personas

Generated by the judge at the start of each round based on spec analysis.

**Language/Stack Engineer** — The judge detects the primary language and technology stack from the spec (Go, Rust, Python, TypeScript, etc.) and generates a persona with idiomatic expertise for that stack. Examples:
- Go project → Go engineer focused on concurrency patterns, interface design, error handling conventions.
- Python ML project → Python engineer focused on type safety, dependency management, ML pipeline patterns.
- Rust CLI → Rust engineer focused on ownership patterns, error handling with `thiserror`/`anyhow`, CLI argument design.

**Domain Expert(s)** — The judge identifies 1-3 domain-specific areas that require specialized review. The number and focus depend on the spec's content. Examples:
- A spec describing a file-based state machine → "Distributed systems engineer" reviewing consistency and crash recovery.
- A spec describing an API → "API design reviewer" checking REST conventions, versioning, backwards compatibility.
- A spec touching authentication → "Identity & access management specialist" reviewing auth flows, token handling, session management.

**Constraints on dynamic personas:**
- Maximum 3 dynamic personas per round (including the language engineer).
- The judge must justify each persona choice with a one-sentence rationale tied to spec content.
- Round 2 may produce different domain personas if the spec changed significantly, but the language engineer persona stays consistent unless the stack changed.

### 4.3 Persona Definition Format

Each persona is defined as a structured config:

```json
{
  "persona_id": "go-engineer",
  "display_name": "Go Engineer",
  "type": "dynamic",
  "perspective": "Idiomatic Go expert focused on concurrency, interface design, error handling, and standard library usage.",
  "review_instructions": "Review the technical spec from the perspective of a senior Go engineer. Focus on: ...",
  "focus_sections": ["2. Architecture", "3. Canonical artifacts", "4. Interfaces"],
  "model_preference": "sonnet",
  "generated_by": "judge",
  "rationale": "Spec targets a Go CLI tool with concurrent task execution."
}
```

## 5. Prompt Construction

### 5.1 Review Prompt Structure

Each reviewer receives a prompt constructed from:

1. **Persona context** — who they are and what perspective to take.
2. **The spec** — full technical spec markdown.
3. **Review instructions** — what to look for, what output format to produce.
4. **Codebase context** (when available) — relevant source files, directory structure, or Roam semantic context if the spec references existing code.
5. **Prior round context** (round 2 only) — the previous round's findings and which were accepted/rejected, so reviewers don't repeat addressed issues.

### 5.2 The Nudge Pass

After each reviewer produces initial findings, the orchestrator sends a follow-up prompt:

> "Good findings so far. We are surprised you did not find more issues. Please review the spec again, paying particular attention to areas you may have skimmed. Focus on edge cases, implicit assumptions, and requirements that are under-specified."

This consistently triggers deeper analysis. The nudge output is appended to the reviewer's findings (not a separate review).

### 5.3 Output Format

Each reviewer produces a structured findings document:

```json
{
  "persona_id": "security-perf-engineer",
  "round": 1,
  "findings": [
    {
      "finding_id": "sec-001",
      "severity": "high",
      "section": "4. Interfaces",
      "location": "Section 4.2, paragraph 3",
      "finding": "The spec does not specify authentication for the inter-worker communication channel.",
      "suggested_change": "Add requirement: all inter-worker messages must be authenticated with HMAC-SHA256 using a per-run shared secret.",
      "rationale": "Without authentication, a malicious process on the same machine could inject fake completion reports."
    }
  ],
  "summary": "Found 3 high, 2 medium, 1 low severity issues. Primary concern is unauthenticated worker communication.",
  "confidence": 0.87
}
```

## 6. Judge/Editor Consolidation

### 6.1 Role

The judge/editor is the most critical agent in the pipeline. It must:

1. **Cross-validate** — Check each finding against the spec and codebase for factual accuracy. Reject findings that are wrong or based on misunderstanding.
2. **Deduplicate** — Multiple reviewers may flag the same issue differently. Group related findings.
3. **Apply** — Modify the spec to address accepted findings. Preserve the spec's voice and structure.
4. **Explain rejections** — Every rejected finding gets a reason: factually incorrect, already addressed, out of scope, conflicts with a design constraint, etc.
5. **Drift-check** — After applying changes, verify no existing content was accidentally dropped. Diff spec v(n) against v(n-1) and flag any removed sections.

### 6.2 Model Requirements

The judge/editor must be an Opus-class model with extended thinking enabled. This role requires:
- Holding the full spec + all findings in context simultaneously.
- Making nuanced judgment calls about finding validity.
- Producing accurate spec edits without introducing new errors.
- Cross-referencing findings against each other for consistency.

### 6.3 Rejection Log Format

```json
{
  "round": 1,
  "rejections": [
    {
      "finding_id": "perf-003",
      "persona_id": "security-perf-engineer",
      "severity": "medium",
      "finding": "The spec should mandate connection pooling for database access.",
      "rejection_reason": "The system uses file-based state (no database). The reviewer appears to have assumed a database-backed architecture.",
      "cross_reference": "See spec section 3.1: 'All state is persisted as JSON files under .attest/runs/'"
    }
  ]
}
```

## 7. Annotation Model

### 7.1 The `@@` Syntax

After the pipeline produces spec v2, the human reviews it. For remaining issues, the human adds inline annotations:

```markdown
## 4. Interfaces

The CLI accepts spec files via `--spec` flags.

@@ This should also support reading from stdin for piped workflows.

Tasks are dispatched via the engine's GetPendingTasks method.
```

### 7.2 Annotation Rules

- `@@` annotations are inline markers in the spec markdown.
- Each annotation is a single line starting with `@@` followed by the comment text.
- Unresolved `@@` annotations block `attest tech-spec approve`.
- Annotations are extracted and presented as review findings during approval checks.
- Once addressed (by editing the spec to resolve the concern), the `@@` line is removed.

### 7.3 Versioning and Drift Detection

Each round produces a versioned spec:
- `technical-spec-v0.md` — human draft
- `technical-spec-v1.md` — post-round-1
- `technical-spec-v2.md` — post-round-2

The judge/editor runs a structural diff between versions and flags:
- Removed sections (possible accidental deletion)
- Removed requirements (may need explicit justification)
- Significantly shortened sections (possible information loss)

The active spec (`technical-spec.md`) is always the latest version. Previous versions are kept for audit.

## 8. Orchestration

### 8.1 Parallel Execution

Reviews within a round execute in parallel using existing CLI tools:
- Claude CLI (`claude`)
- Codex CLI (`codex`)
- Other authenticated CLI tools the user has available

Each reviewer runs as a separate process. The orchestrator:
1. Generates persona configs and prompts.
2. Launches all reviewers in parallel.
3. Waits for all to complete (with per-reviewer timeout).
4. Collects structured findings.
5. Passes all findings to the judge/editor.

### 8.2 CLI Surface

```
attest tech-spec review <run-id>              # runs the full council pipeline (2 rounds)
attest tech-spec review <run-id> --round 1    # run only round 1 (for debugging/testing)
attest tech-spec review <run-id> --dry-run    # generate personas and prompts without executing
```

The existing structural checks become a pre-flight validation before the council runs.

### 8.3 Artifacts Produced Per Round

```
.attest/runs/<run-id>/
  technical-spec-v0.md                    # human draft (input)
  technical-spec-v1.md                    # post-round-1
  technical-spec-v2.md                    # post-round-2
  technical-spec.md                       # active (symlink or copy of latest)
  council/
    round-1/
      personas.json                       # generated persona configs
      review-go-engineer.json             # individual review output
      review-security-perf-engineer.json
      review-testability-reviewer.json
      review-domain-state-machines.json
      consolidation.json                  # judge decision log
      rejection-log.json                  # rejected findings with reasons
    round-2/
      personas.json
      review-*.json
      consolidation.json
      rejection-log.json
  technical-spec-review.json              # final review status (pass/fail)
```

## 9. Integration with Existing Pipeline

### 9.1 Gate Enforcement

The council review pipeline replaces the current structural-only review as the gate for `attest tech-spec approve`. The structural checks (required sections) become a pre-flight that runs before the council.

```
attest tech-spec draft    → records spec v0
attest tech-spec review   → runs council pipeline, produces spec v2 + review artifacts
attest tech-spec approve  → validates: review passed, no unresolved @@ annotations, spec hash matches
```

### 9.2 Downstream Impact

No changes to the downstream pipeline. The execution plan draft still reads from the approved `technical-spec.md`. The difference is that the approved spec is now substantively reviewed, not just structurally checked.

## 10. Implementation Slices

### Slice 1: Single-round review with fixed personas

- Define persona configs for security/perf and testability reviewers.
- Build prompt construction from persona + spec.
- Run reviews sequentially (not parallel yet) using a single CLI tool.
- Output structured findings to JSON files.
- No editor consolidation — human reviews raw findings.
- **Value:** Proves the persona/prompt model produces useful findings.

### Slice 2: Judge/editor consolidation

- Build the judge prompt for cross-validation and spec update.
- Implement rejection logging.
- Implement drift detection (diff v0 → v1).
- Produce spec v1 from round 1 findings.
- **Value:** Closes the loop — findings become spec improvements automatically.

### Slice 3: Dynamic persona generation

- Build the judge prompt for spec analysis and persona generation.
- Detect language/stack from spec content.
- Generate 1-3 domain personas with rationale.
- **Value:** Reviews adapt to the spec's domain without manual configuration.

### Slice 4: Two-round orchestration

- Wire up round 2 with spec v1 + round 1 context as input.
- Implement persona re-evaluation between rounds.
- Implement the nudge pass within each reviewer's execution.
- **Value:** Full two-round pipeline with convergence.

### Slice 5: Parallel execution

- Launch reviewers as parallel processes.
- Per-reviewer timeout handling.
- Support multiple CLI backends (claude, codex).
- **Value:** Wall-clock time drops from sum to max of reviewers.

### Slice 6: Annotation support

- Parse `@@` annotations from spec markdown.
- Block approval on unresolved annotations.
- Surface annotations as findings in the review status.
- **Value:** Human feedback loop integrated into the approval gate.

## 11. Open Questions

1. **Reviewer backend selection:** Should the persona config specify which CLI tool to use (claude vs codex), or should the orchestrator decide based on availability and model preference?

2. **Codebase context scope:** When providing codebase context to reviewers, how much is too much? Full source dump? Only files referenced in the spec? Roam-scoped context?

3. **Spec mutation strategy:** Should the editor apply findings as minimal surgical edits, or rewrite affected sections holistically? Minimal edits are safer but may produce awkward prose. Holistic rewrites risk information loss.

4. **Timeout policy:** What happens when a reviewer times out? Skip it and proceed with available findings, or fail the round?

5. **Cost visibility:** Should the pipeline report estimated token usage before execution so the user can decide whether to proceed?
