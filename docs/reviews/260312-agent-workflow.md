# Agent Workflow Review

Reviewer perspective: CLI-native agent workflow realism, adoption friction, model-routing practicality, multi-agent usability.
Review target: `docs/specs/attest-technical-spec.md` with supporting context from `docs/specs/attest-functional-spec.md`.
Review type: Independent round-1 review.

---

## Findings

- [critical] Detached execution has no survivable process contract, making the core "launch and leave" promise undeliverable
  - Section: 12.1 (Launch model), 12.2 (Reattachment model), 13.1 (Required status fields), 2.1 (Local run engine)
  - Workflow impact: The entire product thesis is "approve once, walk away, come back later." Section 12.1 says the launching session may exit without cancelling the run. Section 12.2 says reattachment reads repo-local state. But there is no specification of what process survives terminal close, how it detaches, what liveness metadata it writes, or how `attest status` distinguishes a live engine from a dead one. This is not a missing detail -- it is the absence of the mechanism that makes the product work.
  - Operator pain: A user launches a run, closes their laptop, reopens it the next morning, runs `attest status`, and sees tasks in `implementing` with active claims. They have no way to know whether the engine is alive and progressing, silently dead with stale state, or stuck behind an expired claim that will never heartbeat again. The rational response is to never trust detached mode, which eliminates the product's primary value. Users will instead babysit runs in a foreground terminal, defeating the purpose.
  - Required correction: Define the v1 run-engine process model explicitly. At minimum: (1) `attest launch` starts a background daemon or long-lived subprocess that detaches from the terminal, (2) the engine writes a liveness artifact (PID file + heartbeat timestamp) under `.attest/runs/<run-id>/`, (3) `attest status` checks liveness by reading the heartbeat and probing the PID, (4) `attest resume` detects a dead engine and takes over after expiring stale claims, (5) the spec states what happens after host reboot (engine is dead, run is resumable from durable state, stale claims expire on next `resume`). Without this, AT-TS-002 ("a launched run persists after the launcher exits") is untestable.

- [critical] No degraded-mode policy for mandatory reviewer backends, making runs hostage to CLI auth and vendor availability
  - Section: 5.3 (Closure rule), 8.2 (Codex backend), 8.3 (Gemini backend), 14.1 (Backend failures), 18 (Open v1 decisions)
  - Workflow impact: The closure rule (5.3) requires Codex review non-blocking AND Gemini review non-blocking. Backend failure handling (14.1) says "review timeout -> blocking review failure unless quorum policy allows downgrade," but the quorum policy is deferred to section 18 as an open decision. This means a single expired Codex API key, a Gemini CLI version mismatch, or a transient rate limit on either service blocks every task from closing -- indefinitely, with no defined recovery path.
  - Operator pain: In real CLI workflows, backend auth issues are routine. A Codex token expires at 2am while the run is detached. Every task that reaches the review gate piles up in `under_review`. The user returns to find zero progress and a wall of reviewer-infrastructure failures, not product failures. The tool feels like it creates more operational overhead than it removes. Worse, because the quorum policy is undefined, there is no documented way for the operator to unblock the run short of fixing the Codex auth and rerunning every stalled review. Users will learn to avoid detached mode or will hack around the review gates.
  - Required correction: Define a v1 degraded-mode matrix: (1) preflight backend availability check at launch time -- warn or block if Codex/Gemini CLI is missing, unauthenticated, or unreachable, (2) per-review-gate retry budget and timeout budget (e.g., 3 retries, 10-minute timeout per attempt), (3) explicit behavior when retries are exhausted -- task moves to `blocked` with a structured reason, not silently stuck in `under_review`, (4) define whether a human can approve reduced-confidence continuation (e.g., `attest override <run-id> <task-id> --skip-codex-review --reason "auth expired, will re-review manually"`), (5) move the quorum decision out of section 18 into a normative rule -- even if the v1 answer is "strict unanimity required, no quorum downgrade."

- [high] The per-task review loop is too expensive by default and has no convergence bound, risking unbounded token spend and churn on routine tasks
  - Section: 5.3 (Closure rule), 9.2 (Escalation rules), 10.2 (Reviewer order), 11.1 (Repair triggers), 18 (Open v1 decisions)
  - Workflow impact: Every task -- regardless of risk, size, or complexity -- must pass through deterministic verifier, Codex review, Gemini review, and Opus synthesis before it can close. When any gate blocks, a repair task is created, and the full four-gate loop runs again. The escalation rules (9.2) mention "repair loops do not converge" as a trigger for Sonnet-to-Opus escalation, but "converge" is undefined and the maximum retry/repair depth is deferred to section 18. This creates a loop with no guaranteed termination.
  - Operator pain: A task with a subtle spec ambiguity fails Gemini review. A repair task is created. The repair addresses Gemini's concern but introduces a style issue that Codex flags. Another repair task. Each iteration costs a full Opus synthesis call plus two independent reviews. After five iterations the user has spent significant tokens and wall-clock time on what started as a low-risk utility function. The run never reaches `completed` because the repair loop never converges -- different reviewers keep finding different marginal issues. The user's trust in the tool's cost-effectiveness collapses.
  - Required correction: (1) Make full council review risk-gated, not universal. Low-risk tasks (as classified by the task compiler) should require verifier + one reviewer + lightweight synthesis. High-risk tasks get the full panel. (2) Define a maximum repair depth per task lineage (e.g., 3 repair attempts). Exceeding the depth transitions the root task to `blocked` with a structured explanation. (3) Require each repair task to be strictly narrower than the finding that triggered it -- no reopening the full task scope on a marginal finding. (4) Fingerprint findings so that a reviewer restating a previously-addressed concern does not trigger a new repair cycle. (5) Move the retry/depth ceiling out of section 18 into a normative rule.

- [high] `attest resume` is a command with no defined semantics, creating a critical gap in the operator's recovery workflow
  - Section: 4.1 (Core commands), 12.2 (Reattachment model), 5.1 (Run states)
  - Workflow impact: `attest resume <run-id>` is listed as a command. Section 12.2 says reattachment reads repo-local state. But `attest status` already covers observation. What does `resume` actually do? Does it restart a dead engine? Retry blocked tasks? Prompt for pending clarifications? Work on a `stopped` run? A `failed` run? A `blocked` run? Each of these requires different behavior, and none is specified.
  - Operator pain: The user's run engine crashed overnight. They run `attest resume`. If the implementer interpreted `resume` as "display status" (since that is all the spec supports), the user expects a restart but gets a status dump. If `resume` does restart the engine, it needs to expire stale claims first -- but that logic is not specified. If the run is in `stopped` state, is `resume` or `launch` the right command? The user guesses wrong, gets an error, and loses trust in the command surface. This is the single most important recovery command and it has zero defined behavior.
  - Required correction: Define `resume` as: (1) detect whether the run engine is alive via the liveness artifact, (2) if dead, expire all stale claims, restart the engine, and report what was recovered, (3) if alive, reattach as an observer and report current status, (4) if the run is in `blocked` state, present the pending blockers and accept user input to resolve them, (5) specify which run states are resumable (`running`, `blocked`, `stopped`) and which are terminal (`completed`, `failed`). Make `resume` the single command a returning operator needs.

- [high] Status and explainability are too thin to answer the operator's actual question: "what is it doing right now and why?"
  - Section: 13.1 (Required status fields), 13.2 (Explainability requirements), 4.1 (`attest explain`)
  - Workflow impact: `run-status.json` exposes aggregate counts (task counts by state, active claims, uncovered requirement count) and timestamps (last successful council, last failure). `attest explain` must answer "why not finished" and "what next action is." But neither artifact specifies per-task operational detail: which gate is this task waiting at, which backend is active, how long has it been waiting, what is the retry count, what specific finding is blocking it, and what will happen next automatically versus what requires human action.
  - Operator pain: The user runs `attest status` and sees "running, 3 tasks implementing, 2 under_review, 1 blocked." This tells them almost nothing actionable. They need to know: "task-7 is waiting for Codex review (attempt 2 of 3, started 4 minutes ago), task-12 is blocked because Gemini flagged finding F-034 (missing null check in auth handler), task-3 is implementing via Sonnet (claimed 12 minutes ago, heartbeat 30 seconds ago)." Without this, users will browse `.attest/runs/<run-id>/reports/` manually, which breaks the promise of CLI-native inspection. The tool becomes a file-system layout convention instead of a useful status surface.
  - Required correction: Require per-active-task status fields: current gate (implementing/verifying/codex-review/gemini-review/opus-synthesis), active backend and model, claim age, last heartbeat age, retry count, next automatic action, blocking finding reference (finding ID + one-line summary), and whether the blocker requires human input. `attest explain` should render this per-task detail, not just aggregates.

- [high] Codex and Gemini review input is unbounded, making review quality unpredictable and cost uncontrollable
  - Section: 8.2 (Codex backend), 8.3 (Gemini backend), 9.1 (Default routing table), 10.2 (Reviewer order)
  - Workflow impact: The spec assigns Codex and Gemini specific review roles (code review, spec review, test adequacy, etc.) but never defines what input each reviewer receives. Does Codex get the full repo? The full diff? The completion report? The entire run artifact? The requirement text? All prior findings? The answer determines whether review is fast, focused, and affordable, or slow, noisy, and expensive.
  - Operator pain: If the review packet is too large, Codex and Gemini will produce unfocused reviews that flag irrelevant issues outside the task scope, driving unnecessary repair cycles. If the packet is too small, they will miss context and produce shallow reviews that add cost without adding signal. If the packet varies per backend because no standard is defined, review quality will be inconsistent and hard to debug. The operator cannot predict or control review cost because the input size is unbounded.
  - Required correction: Define a standard review bundle per task attempt: (1) the task scope and linked requirement subset from the run artifact, (2) the actual file diff (not the full repo), (3) the completion report, (4) the verifier result, (5) prior blocking findings for this task only, (6) a context budget (e.g., maximum token count or file count) beyond which the review escalates to a more capable model tier rather than stuffing the context window. Larger context should be an explicit escalation, not the default.

- [medium] The CLI command surface has ambiguous boundaries that will confuse everyday use
  - Section: 4.1 (Core commands), 4.2 (Agent-session integration)
  - Workflow impact: `review`, `approve`, `launch`, `status`, `explain`, and `resume` are all listed but their distinctions are not operationally sharp. Is `review` pre-approval artifact inspection or reviewer rerun? Why are `approve` and `launch` separate commands when the primary journey (functional spec 9.1) does both back-to-back? How does `status` differ from `explain`? Can `status` with no run-id list all runs, or does it require a run-id?
  - Operator pain: Users working inside Claude Code, Codex CLI, or Gemini CLI sessions will ask the agent "show me the run status" and the agent will not know whether to call `attest status`, `attest explain`, or read the status file directly. The agent-session integration (4.2) says the agent is responsible for calling these commands, but the agent cannot reliably choose between ambiguous commands. Users will develop ad-hoc habits instead of a clean workflow.
  - Required correction: (1) Make `review` explicitly pre-approval only (inspect the run artifact before approving), (2) provide `approve --launch` as the default combined path, with `approve` alone for users who want to inspect the task graph before launching, (3) define `status` with no run-id as a run-listing command, `status <run-id>` as the quick summary, and `explain <run-id>` as the detailed diagnostic, (4) define `resume` as the single recovery/reattach action command (as specified in the resume finding above).

- [medium] No preflight validation at launch time means failures will surface late and confusingly
  - Section: 4.1 (`attest launch`), 8.1-8.3 (Execution backends), 14.1 (Backend failures)
  - Workflow impact: `attest launch` transitions the run to detached execution. But there is no specified preflight check for: backend CLI binaries installed and on PATH, backend authentication valid, required model tiers accessible, disk space available, or conflicting active runs in the same repo. Section 14.1 handles "missing CLI binary" as a runtime failure, but by then the run is detached and the user may have left.
  - Operator pain: The user approves a run, launches it, closes the terminal. Ten seconds later the Codex CLI binary is not found. The first task completes implementation but blocks at review because Codex is unavailable. The user returns hours later to find zero completed tasks and a cryptic backend-unavailable blocker. A 10-second preflight check at launch time would have caught this immediately. Failing to validate upfront converts cheap, fixable configuration problems into expensive, confusing runtime failures.
  - Required correction: Require `attest launch` to perform preflight validation before transitioning to detached execution: (1) all mandatory backend CLIs are present and authenticated, (2) a trivial health-check call to each backend succeeds, (3) no other run in `running` state exists for this repo (or define the multi-run isolation model), (4) the run artifact hash matches the approved hash. Preflight failures should block launch with a clear diagnostic, not silently start a doomed run.

- [medium] `attest stop` has no defined behavior, making it dangerous to use
  - Section: 4.1 (Core commands), 5.1 (Run states)
  - Workflow impact: `stop` exists as a command and `stopped` exists as a run state, but there is no specification of whether stop is graceful or immediate, what happens to in-flight workers and their claims, whether a stopped run can be resumed, or what state in-flight tasks transition to.
  - Operator pain: A user needs to stop a run to fix a spec issue. They run `attest stop`. If stop kills the engine immediately, in-flight workers may be orphaned -- they continue running detached, consuming API tokens, and writing reports that no one will process. If stop is graceful with a long grace period, the user waits an unknown amount of time. If the stopped run cannot be resumed, the user must re-approve and re-launch, losing all completed work. A stop command that is semantically fuzzy creates fear of data loss and orphaned processes, so users will avoid using it.
  - Required correction: Define stop behavior: (1) `attest stop` signals the engine to stop dispatching new work, (2) in-flight workers are given a configurable grace period to complete (default: current lease duration), (3) after the grace period, remaining claims are expired, (4) the run transitions to `stopped`, (5) `attest resume` can restart a stopped run. Optionally support `attest stop --force` which kills workers immediately.

- [medium] Fresh-session run discovery requires the user to already know the run-id
  - Section: 4.1 (Core commands), functional spec 8.7 (AT-FR-030)
  - Workflow impact: AT-FR-030 says "a later agent session must be able to list active runs." But the CLI surface only defines `attest status [<run-id>]`. There is no explicit list command or no-arg behavior that shows all runs in the current repo.
  - Operator pain: A user returns to a repo days later and does not remember the run-id. They type `attest status` with no arguments. If this shows nothing useful, they must manually browse `.attest/runs/` to find run directories. If they have multiple runs (some completed, some stopped, one running), there is no way to see a summary without knowing each run-id. This is the first command a returning operator will reach for, and it does not work without prior knowledge.
  - Required correction: Define `attest status` with no run-id as listing all runs in the current repo, sorted by last activity, showing run-id, state, task progress summary, and last update time. If only one run exists, default to showing that run's detail.

- [medium] The approval artifact is raw normalized JSON, which operators will not meaningfully review
  - Section: 3.2 (`run-artifact.json`), 4.2 (Agent-session integration), functional spec 7.1 (Planning and approval)
  - Workflow impact: The approval boundary is the most important human checkpoint in the system. But the only defined artifact is `run-artifact.json` -- a machine-oriented JSON file with requirements, assumptions, clarifications, dependencies, risk profile, and routing policy. The spec expects the agent session to "present the run artifact" and "collect human approval," but provides no human-readable approval summary format.
  - Operator pain: The agent presents a 200-line JSON file. The user skims it, says "looks fine," and approves. The approval boundary -- which is supposed to prevent bad task decomposition and missed requirements -- becomes a rubber stamp because the artifact is not designed for human consumption. Alternatively, the agent summarizes the JSON in its own words, and the user approves the summary without seeing the actual artifact, which means they are approving the agent's interpretation, not the contract.
  - Required correction: Require `attest review <run-id>` to produce a structured human-readable summary: (1) requirement count and IDs, (2) assumptions made, (3) open clarifications, (4) dependency chain, (5) risk classification, (6) proposed task breakdown preview, (7) source-spec fingerprint so the operator knows exactly what source material was ingested. The JSON artifact remains the machine contract; the summary is the approval surface.

- [low] Routing traceability records decisions but not the inputs that produced them
  - Section: 9.3 (Routing traceability)
  - Workflow impact: Every task attempt records selected backend, model tier, escalation cause, and retry count. But it does not record the prompt template version, input bundle hash, or backend CLI version. When a review produces a different result on retry, the operator cannot tell whether the cause was a model change, a prompt change, or a different evidence bundle.
  - Operator pain: Minor in v1, but becomes a real debugging problem as soon as the user needs to understand why two review attempts on the same task produced contradictory findings. Without input identity, routing experiments and escalation analysis are guesswork.
  - Required correction: Add to the traceability record: backend CLI version, prompt/template identifier, and input bundle content hash.

- [low] Reviewer findings have no stable IDs, making repair tracking and deduplication noisy
  - Section: 3.5 (Task report files -- codex-review.json, gemini-review.json, council-result.json)
  - Workflow impact: Reviewer outputs contain blocking and non-blocking findings, but there is no normalized finding ID, severity level, category, or deduplication key. When a finding from round 1 is restated in round 2 (possibly in different words), the system has no way to recognize it as the same finding.
  - Operator pain: The user sees "3 blocking findings" on a task and cannot tell whether they are 3 new issues or 1 issue restated 3 times across rounds. Repair tasks are created for each instance, leading to redundant work. `attest explain` cannot say "this is the same issue that was flagged in round 1" because findings have no identity.
  - Required correction: Add to reviewer output schemas: `finding_id`, `severity`, `category`, `evidence_ref`, and `dedupe_key`. The repair reconciler should match new findings against prior findings by dedupe_key before creating repair work.

- [low] Completion reports do not include enough operational metadata for quick operator trust assessment
  - Section: 3.5 (Task report files -- completion-report.json)
  - Workflow impact: The completion report lists commands executed and tests executed, but not per-command exit codes, durations, working directories, or whether each command was required or optional.
  - Operator pain: The operator sees "commands executed: [build.sh, test.sh]" and has to trust that both succeeded. If build.sh failed but the implementer moved on, the report does not surface this. The operator must dig into raw logs to verify, which is exactly the kind of manual inspection the tool is supposed to eliminate.
  - Required correction: Require per-command entries in the completion report to include: command string, exit code, duration, working directory, and required/optional classification.

## Adoption Risks

- The single largest adoption risk is trust loss on reattach. If users cannot tell whether a detached run is alive, advancing, or silently dead, they will stop using detached mode within a week. The tool becomes a complicated foreground runner, which is not differentiated from existing agent CLI workflows.
- Mandatory reviewer gates without explicit degraded-mode handling will make the tool feel hostage to vendor infrastructure. The first time a Codex auth token expires overnight and blocks an entire run, the user will question whether multi-model review is worth the operational tax.
- Ambiguous command semantics (`review` vs `status` vs `explain`, `resume` vs `launch`) will push users back toward ad-hoc conversational orchestration ("just read the files and tell me what happened") instead of using the product's intended command surface.
- The approval artifact problem is subtle but corrosive. If operators cannot meaningfully review what they are approving, the approval boundary degrades from a safety checkpoint to a speed bump. The system's traceability story weakens because the human never actually verified the contract.
- The absence of preflight validation means the first experience for many users will be: approve a run, launch it, leave, come back to find it failed on a trivially fixable configuration issue. First impressions of an autonomous tool are disproportionately important for trust formation.

## Routing And Cost Risks

- The default routing policy (section 9.1) is directionally correct -- Haiku for scouting, Sonnet for implementation, Opus for synthesis. The problem is the review pipeline, not the routing table. Requiring verifier + Codex + Gemini + Opus synthesis on every single task, regardless of risk level, creates a per-task cost floor that is too high for routine work items.
- A run with 20 tasks, each requiring 2-3 review rounds to converge, will invoke Opus synthesis 40-60 times, Codex review 40-60 times, and Gemini review 40-60 times. Even with per-call costs in the single-digit dollar range, total review cost for a moderate run could easily exceed $100-200 in API spend. This is not unreasonable for high-value work, but it is unreasonable when half those calls are reviewing trivial tasks that a single reviewer could have cleared.
- The escalation rules (9.2) are good in principle but vague in practice. "Repair loops do not converge" needs a concrete definition (e.g., same verifier failure class on consecutive attempts, or N repair attempts without the finding set shrinking). Without this, the escalation trigger is itself an LLM judgment call, which means escalation is non-deterministic and hard to predict.
- Reviewer disagreement as a convergence mechanism only works if findings are fingerprinted and deduplicated. Without finding IDs and dedupe keys, the system cannot distinguish "Codex raised a new issue" from "Codex restated its previous issue in different words." The former should create repair work; the latter should not. The current spec treats both identically, which amplifies cost.
- The review packet problem (unbounded reviewer input) is a cost multiplier. If Codex receives the full repo context on every review call, the per-call token cost is dramatically higher than if it receives a bounded diff + requirement subset. The spec must define review bundle boundaries or cost will be unpredictable.

## What Is Strong

- The approved `run-artifact.json` as a frozen execution contract is the right architectural boundary. Separating free-form preparation from deterministic post-approval execution is exactly right for CLI-native workflows where the user wants to iterate loosely during planning but needs stability during execution.
- The closure rule (5.3) -- requiring deterministic verification AND independent Codex review AND independent Gemini review AND Opus synthesis -- is genuinely strong anti-false-completion design. The rule itself is not the problem; the problem is applying it uniformly to all tasks without risk-gating. The philosophy is correct; the policy needs calibration.
- Keeping v1 local-first and single-binary (Go) is the right distribution decision for CLI-native adoption. No server to run, no Docker container, no cloud account -- just a binary on PATH next to `claude`, `codex`, and `gemini`. This is the only shape that will feel native inside agent sessions.
- The fail-closed principle (7.3, 14.2) -- timeout produces failure, missing report produces failure, malformed output produces failure -- is exactly correct for an autonomous system. False negatives (keeping work open unnecessarily) are much cheaper than false positives (claiming success incorrectly) in this domain.
- Using Codex and Gemini as independent reviewers rather than co-implementers in v1 (1.3) is a smart scope decision. It captures the disagreement value of multi-model review without the coordination complexity of multi-model implementation. This avoids an entire class of merge, prompt, and output-format problems.
- The artifact-per-task report structure (3.5) -- separate files for completion, verification, and each reviewer's output -- is well-decomposed for both machine processing and human inspection. It makes the system auditable at the individual-finding level.
- The reattachment-from-repo-local-state constraint (12.2) is the right foundation. No transcript dependency, no remote state, no session affinity. If the process model is defined (per the critical finding above), this constraint makes everything else work.
